# Parser-Assisted Chunking Experiment

This directory is the entry point for the parser-assisted chunking experiment design.

The experiment asks whether conventional linguistic parsing plus deterministic RSVP boundary optimization can reduce harmful grammatical and semantic splits while preserving the current reader's display constraints, source coverage, timing contract, and fallback behavior.

## Current Production Baseline

The production baseline remains the dependency-free rule-based chunker:

- Baseline commit: `8b50a3049bb5d92a304a03527385c519194ce8da`
- Chunker entry point: `semantic_rsvp.chunking.rules.RuleBasedChunker`
- Normalization entry point: `semantic_rsvp.text.normalize.normalize_text`
- Sentence/source-boundary entry point: `semantic_rsvp.text.segment.split_sentences`
- Schedule entry point: `semantic_rsvp.timing.schedule.schedule_text`
- Default chunker configuration: `max_chars=32`, `max_content_words=2`

The current chunker remains the production default, regression baseline, fallback implementation, and comparison target.

## Proposed Future Architecture

The future experimental path is:

1. A local parser adapter extracts fallible linguistic evidence into project-owned records.
2. A deterministic boundary optimizer combines that evidence with RSVP constraints.
3. The existing timing and schedule layer remains downstream and unchanged.

Parser output is evidence, not grammatical truth. RSVP policy remains project-owned: display width, punctuation behavior, source structure, timing compatibility, visual balance, and fallback behavior are not delegated to a parser.

## Parser-Assisted Spike Operation

S-023 adds an isolated experimental parser-assisted chunker under `semantic_rsvp.experiments.parser_assisted_chunking`. It is not a production default and is only selected by explicit imports or the experiment runner.

Install the optional NLP environment separately:

```bash
python -m venv .venv-nlp
.venv-nlp\Scripts\python.exe -m pip install -r requirements-nlp-spike.txt
```

Run the optional integration tests and visible-corpus experiment:

```bash
.venv-nlp\Scripts\python.exe -m pytest tests\test_parser_assisted_spacy_integration.py
.venv-nlp\Scripts\python.exe scripts\run_parser_assisted_experiment.py --corpus development --corpus regression --write
```

The runner writes stable side-by-side diagnostics to `evaluation/parser_assisted_chunking/results/parser_assisted_development_regression.json`. Use `--include-latency` for local timing measurements; latency values are intentionally omitted from the committed reproducible result file.

The optimizer uses project-owned token, span, relation, parse, and trace records. spaCy objects do not leave the adapter layer. The `max_content_words=2` policy is preserved as a high-cost RSVP density constraint rather than a hard rejection, while `max_chars=32` remains a hard width limit except for explicitly marked unsplittable-token cases.

## Safe And Unsafe Materials For Codex

Safe for Codex during implementation:

- Development corpus inputs and future development annotations.
- Frozen regression corpus inputs and baseline outputs.
- Frozen generalization inputs.
- Baseline outputs and evaluation scripts in `evaluation/parser_assisted_chunking/`.

Must remain unavailable during tuning:

- Human-held blind challenge source text.
- Human-held blind challenge annotations.
- Frozen generalization gold annotations, if the human creates them outside the repository.

## Reproducing Baseline Artifacts

```bash
python scripts/validate_chunking_corpus.py
python scripts/freeze_chunking_baseline.py --write
python scripts/freeze_chunking_baseline.py --check
```

`--check` fails if re-running the production baseline would change the committed baseline output.

## Comparison Summary

Future results should compare baseline and experiment outputs using:

- Hard compliance: source coverage, structural boundaries, maximum width, alignment, fallback rate.
- Linguistic quality: forbidden-boundary violations, protected-span violations, required-boundary recall, recurrence of known defects, new regressions, over-clumping.
- Distributional behavior: chunk lengths, words per chunk, single-word rate, long-chunk rate.
- Operational cost: latency, parser/model footprint, fallback percentage.
- Human comparison: blinded A/B preference with `A`, `B`, `equivalent`, and `both poor` options.

See [Evaluation Protocol](evaluation_protocol.md) and [Experiment Contract](experiment_contract.md).

For contamination rules and blind-set checksum handling, see [Contamination Policy](contamination_policy.md).
